# SINTOMAS OBSERVADOS

## Pool starvation

Erros recorrentes:

sqlalchemy.exc.TimeoutError:
QueuePool limit of size 5 overflow 5 reached

## Sessões ruins

Encontrado em pg_stat_activity:

* idle in transaction
* transações abertas por 20s+
* múltiplas sessões active simultaneamente
* conexões não retornando rapidamente ao pool

## Pipeline degradado

Dashboard operacional confirmou:

* ingestão 15m parada
* score engine parado
* celery backlog acumulando
* workers congestionados
* alpha_scores sem atualização
* decisions_log sem novas decisões

## Queries observadas

Transações abertas envolvendo:

INSERT INTO ohlcv(...)
INSERT INTO market_metadata(...)
SELECT FROM reconciled_gate_trades(...)

## Lock contention

Existem RowExclusiveLocks normais, mas NÃO há deadlock massivo.
O problema principal parece ser:

* transações longas
* persistência concorrente excessiva
* lifecycle incorreto de AsyncSession
* compute acoplado à persistência
* ausência de backpressure

# DIAGNÓSTICO CONFIRMADO

O sistema sofre de:

write concurrency amplification

causado por:

* writes concorrentes distribuídos
* fan-out sem controle
* sessões compartilhadas ou long-lived
* commit tardio
* persistência dentro de loops longos
* await externo com transação aberta
* scheduler produzindo mais rápido do que o banco consegue drenar

O problema NÃO deve ser resolvido apenas aumentando pool_size/max_overflow.

# OBJETIVO DA REFATORAÇÃO

Implementar uma arquitetura definitiva baseada em:

* persistência desacoplada
* backpressure explícito
* bounded queues
* persistence workers fixos
* session-per-unit-of-work
* transações curtas
* Unit of Work
* repositories
* retry controlado
* idempotência
* controle explícito de concorrência

# ARQUITETURA DESEJADA

## Fluxo correto

Collectors/Schedulers
→ Compute
→ Async Queue
→ Persistence Workers limitados
→ Postgres

## Regras obrigatórias

* scheduler NÃO escreve mais diretamente no banco
* compute NÃO mantém transação aberta
* cada persistência usa sua própria AsyncSession
* commit imediato após write
* rollback obrigatório
* close obrigatório
* transações devem durar milissegundos
* nenhuma chamada HTTP/Redis/API externa dentro da transação
* nenhuma sessão compartilhada entre coroutines
* eliminar asyncio.gather com writes usando shared session
* limitar concorrência de persistência explicitamente

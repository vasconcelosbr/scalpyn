O Scalpyn tem um pipeline L1→L2→L3 que classifica símbolos por nível.
Um símbolo é considerado APROVADO REAL para execução quando:
  decisions_log.decision = 'ALLOW'  AND  decisions_log.level_direction = 'up'

  level_direction = 'up'   → símbolo SUBIU para L3 (promovido de L2) — é o aprovado real
  level_direction = 'down' → símbolo SAIU de L3 (rebaixado) — não executa

Quando uma promoção (up) chega na execução mas não vira trade por falta de saldo USDT
(capital preso em posições abertas aguardando venda), some sem rastro.
Essas promoções sem execução são oportunidades perdidas que hoje somem — não deixam
rastro no banco, não alimentam o ML, e não são visíveis para o usuário.

Este prompt implementa o **Shadow Portfolio**: quando não há saldo, o sistema
acompanha o ativo como se tivesse comprado U$ 1.000 USDT, usando os mesmos
TP/SL do config real, até atingir TP, SL ou timeout — e registra o resultado
separadamente, sem contaminar nenhum indicador de performance real.

**Princípios que não podem ser violados:**
- ZERO HARDCODE — nenhum valor de negócio fixado no código; tudo via config_service ou parâmetro
- ESTRITAMENTE ADITIVO — nenhuma rota, tabela, coluna ou função existente é removida
- Shadow data nunca contamina métricas reais (P&L, win rate, posições, capital)
- Leia todos os arquivos relevantes ANTES de escrever qualquer código
- Edite cirurgicamente — nunca reescreva arquivos do zero
- Mostre diff antes de aplicar cada mudança
- Imports pesados (xgboost, shap, sklearn) sempre lazy dentro dos handlers

## ARQUITETURA DO SHADOW PORTFOLIO

```
L3: decision='ALLOW' AND level_direction='up'
    ↓
execute_buy.py — gate de capital (SpotCapitalManager)
    ├── TEM SALDO → fluxo real (não muda nada)
    └── NÃO TEM SALDO
            ↓
        Cria shadow_trades (nova tabela)
        amount_usdt = 1000.0 (fixo — ver §1.1)
        entry_price = open da próxima candle 1m após a promoção (level_direction='up')
        tp_pct / sl_pct = config real do usuário no momento
        status = PENDING
            ↓
        Celery task: shadow_trade_monitor (a cada 5min)
            ↓
        Itera candles 1m via OHLCV
        Convenção: SL verificado antes de TP na mesma candle
            ↓
        TP atingido → outcome = WIN,  pnl_pct = +tp_pct
        SL atingido → outcome = LOSS, pnl_pct = -sl_pct
        Timeout     → outcome = TIMEOUT, pnl_pct = (exit_price - entry) / entry
            ↓
        status = COMPLETED
            ↓
        SimulationService.record_shadow_as_simulation()
        → grava em trade_simulations com is_simulated=TRUE
          e decision_type='ALLOW' (representa promoções up — compatível com DatasetBuilder)
```

## FASE 3 — CELERY TASK: MONITOR DE SHADOW TRADES

### 3.1 — Criar `backend/app/tasks/shadow_trade_monitor.py`

Task Celery que roda a cada 5 minutos.

```python
@celery_app.task(name="tasks.shadow_trade_monitor")
async def shadow_trade_monitor():
    """
    Itera shadow_trades com status PENDING ou RUNNING.
    Para cada um, avança a simulação candle a candle (1m).
    Quando atinge TP, SL ou timeout: marca COMPLETED e grava em trade_simulations.
    """
```

Lógica por shadow trade:
1. Buscar candles 1m de `ohlcv` após `entry_price_timestamp` (ou última candle processada)
2. Para cada candle, verificar (nesta ordem — SL primeiro, convenção conservadora):
   - `candle.low <= sl_price` → outcome = LOSS
   - `candle.high >= tp_price` → outcome = WIN
   - `candles_processadas >= timeout_candles` → outcome = TIMEOUT
3. Se outcome encontrado:
   - `exit_price` = sl_price, tp_price, ou close da candle de timeout
   - `pnl_pct = (exit_price - entry_price) / entry_price * 100`
   - `pnl_usdt = amount_usdt * pnl_pct / 100`
   - `holding_seconds` = segundos entre entry e exit
   - `status = COMPLETED`, `completed_at = now()`
   - Chamar `record_as_simulation(shadow_trade, db)`
4. Se não encontrado ainda: `status = RUNNING`, salvar progresso

**Processamento em batch:**
- Processar no máximo 50 shadow trades por execução da task
- Logar progresso: `Shadow monitor: N processed, M completed, K errors`

### 3.2 — Método `record_as_simulation()`

Adicionar em `ShadowTradeService`:

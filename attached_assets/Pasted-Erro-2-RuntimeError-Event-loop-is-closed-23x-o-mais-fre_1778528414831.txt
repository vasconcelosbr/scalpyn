Erro 2 — RuntimeError: Event loop is closed (23x — o mais frequente)
Arquivo: SQLAlchemy → asyncpg → connection.close(timeout=2) tentando criar uma task num event loop já fechado.
O que é: Quando uma conexão do pool SQLAlchemy é terminada (do_terminate), o asyncpg tenta cancelar comandos pendentes criando uma asyncio task — mas o event loop já foi fechado. Isso acontece no teardown.
Causa provável: O Celery worker executa tasks síncronas usando loop.run_until_complete() (visível no Erro 3 — collect_market_data.py:146). Quando o loop termina, o pool SQLAlchemy ainda tem conexões abertas tentando se fechar graciosamente num loop que não existe mais.
Fix: No collect_market_data.py, após o run_until_complete, fechar explicitamente o pool antes de o loop ser encerrado:
python# antes de fechar o loop
await engine.dispose()
Ou usar NullPool no contexto Celery para evitar pooling persistente em workers síncronos.

Erro 3 — PendingRollbackError (3x)
Arquivo: database.py:370 → run_db_task → session.begin().__aexit__
O que é: SQLAlchemy tentando fazer commit numa sessão que está num estado inválido (precisa de rollback primeiro). Classicamente causado por uma exceção não tratada dentro de um bloco session.begin() que deixou a transação "suja".
Causa provável: Provavelmente consequência em cascata do Erro 1 ou 2 — um GateAPIError é levantado dentro de um bloco de transação, a sessão entra em estado de erro, e na próxima tentativa (retry ou re-uso da sessão) o SQLAlchemy recusa o commit.
Fix: Em run_db_task (database.py ~370), garantir que qualquer exceção dentro do async with session.begin() faça rollback explícito antes de re-raise:
pythontry:
    async with session.begin():
        return await task(session)
except Exception:
    await session.rollback()
    raise

Erro 4 — InvalidToken / cryptography.fernet.InvalidToken (3x) ⚠️
Arquivo: trade_sync_service.py:46 → encryption.py:42 → fernet.decrypt(token)
O que é: Fernet não consegue descriptografar a API key armazenada no banco. Isso significa que o token criptografado no banco não corresponde à FERNET_KEY atualmente em uso no Cloud Run.
Isso é crítico. Há três possíveis causas:

A FERNET_KEY foi rotacionada no Cloud Run sem migrar os dados existentes no banco.
A chave foi armazenada originalmente com uma key diferente (ambiente local vs. produção).
O dado no banco está corrompido.

Fix imediato: Verificar se a FERNET_KEY no Cloud Run (variável de ambiente) bate com a que foi usada quando as API keys foram salvas. Se houve rotação, implementar MultiFernet para suportar múltiplas keys durante a migração.

Fase 0 — Mapeamento
Prompt de auditoria do pipeline spot + futures

6 blocos de perguntas para mapear o codebase antes de implementar observabilidade

Bloco 1 — Estrutura do pipeline de decisão

Mapear o fluxo real de aprovação L1 → L3

1.1
Qual arquivo/serviço executa a leitura dos ativos aprovados em L3? É o
signal_engine.py
,
score_engine.py
ou outro?
1.2
O pipeline L1→L2→L3 é sequencial (síncrono) ou paralelo (asyncio tasks/Celery)?
1.3
Existe uma função central de "aprovação de trade" ou a decisão é distribuída em múltiplos serviços?
1.4
Qual é o gatilho que inicia o ciclo de avaliação — Celery beat, WebSocket event, polling, ou chamada direta do endpoint?
1.5
Spot e futures usam o mesmo pipeline de decisão ou são totalmente separados? Se separados, quais arquivos diferem?
Bloco 2 — Regras L3 e motivo de reprovação

Entender como as regras são avaliadas e se há registro de falhas

2.1
As regras L3 são loop de condições sobre o JSONB do
config_profiles
ou são funções Python hardcoded?
2.2
Quando uma regra L3 falha, o sistema registra qual regra falhou e qual valor estava fora da faixa, ou apenas descarta o ativo silenciosamente?
2.3
Existe algum campo de "motivo de reprovação" sendo salvo no banco hoje? Se sim, em qual tabela/coluna?
2.4
Cole um exemplo real (ou resumido) da lógica L3 atual — seja o trecho de código ou o JSON de config que define as regras.
2.5
Para futures: a decisão LONG/SHORT acontece dentro da L3 ou é uma etapa separada posterior à aprovação?
Bloco 3 — Execução e envio para exchange

Mapear o caminho da ordem da decisão até a Gate.io

3.1
Qual função/método do
execution_engine.py
envia a ordem para a Gate.io? Cole a assinatura.
3.2
O payload enviado para a exchange é logado hoje? Se sim, em qual nível (DEBUG/INFO) e com qual estrutura?
3.3
A resposta HTTP da exchange (status code, order_id, body) é capturada e persistida ou apenas consumida em memória?
3.4
Existe tratamento de erro diferenciado para rejeição da exchange (ex: saldo insuficiente, par inválido) vs erro de rede (timeout, connection reset)?
3.5
Após envio da ordem, o sistema faz polling ou usa WebSocket para confirmar o preenchimento (filled) da ordem?
Bloco 4 — Persistência e estado atual dos logs

Avaliar o que já existe antes de duplicar esforço

4.1
O projeto usa
structlog
,
logging
padrão do Python, ou outra solução? Existe um
logger
central configurado?
4.2
Existe algum
trace_id
,
correlation_id
ou
request_id
sendo propagado hoje? Em qual camada?
4.3
A tabela
orders
está sendo usada ativamente? O campo
exchange_order_id
é preenchido após confirmação?
4.4
O campo
indicators_at_entry
(JSONB na tabela
trades
) está sendo preenchido com snapshot completo ou está vazio/parcial?
4.5
Existe variável de ambiente
DEBUG
,
LOG_LEVEL
ou similar controlando o nível de verbosidade dos logs?
Bloco 5 — Concorrência e estado assíncrono

Identificar pontos de falha silenciosa por race conditions

5.1
O pipeline de trade usa
asyncio
puro, Celery workers, ou uma combinação? Quantos workers simultâneos podem processar ordens?
5.2
Existe algum mecanismo de lock (Redis lock, DB advisory lock, Semaphore) para evitar que o mesmo ativo seja comprado duas vezes simultaneamente?
5.3
Já ocorreram erros de
PendingRollbackError
ou
DetachedInstanceError
do SQLAlchemy em produção?
5.4
O WebSocket de mercado (preços/scores em tempo real) está na mesma instância do Cloud Run que o motor de execução, ou são processos separados?
5.5
Quando o Cloud Run escala para zero e reinicia, o pipeline retoma automaticamente ou precisa de intervenção manual?
Bloco 6 — Evidências e arquivos-chave

Coletar o codebase real para análise antes de gerar qualquer código

6.1
Cole o conteúdo atual (ou o trecho principal) de
signal_engine.py
— especialmente a função que decide se um ativo passa para execução.
6.2
Cole o conteúdo atual (ou trecho principal) de
execution_engine.py
— a função que monta e envia a ordem.
6.3
Cole o conteúdo atual de
gate_adapter.py
— o método que faz o POST para a Gate.io API v4.
6.4
Existe algum arquivo de task Celery (em
tasks/
) que orquestra o ciclo spot/futures? Se sim, cole o conteúdo de
evaluate_signals.py
.
6.5
Há algum middleware FastAPI, decorator ou context manager já implementado que envolva as chamadas de trade? Cole se existir.

🔴 PROBLEMA 1 — spot_scanner.py com conexão DB fechada (CRÍTICO)
O que é: 3 ERRORs simultâneos às 12:13:05Z originados de spot_scanner.py linha 148:
sqlalchemy.exc.InterfaceError: cannot call Transaction.rollback(): 
the underlying connection is closed
Stack trace completo:
/app/app/engines/spot_scanner.py, line 148, in _loop
    async with AsyncSessionLocal() as db:
Diagnóstico: O spot_scanner está no loop interno (_loop) e ao finalizar o async with AsyncSessionLocal(), o SQLAlchemy tenta fazer rollback mas a conexão asyncpg já foi fechada pelo pool (timeout de conexão ociosa do PostgreSQL ou do Cloud SQL Proxy).
Causa raiz mais provável: O pool de conexões asyncpg tem pool_recycle ou pool_timeout menor que o intervalo entre iterações do scanner. A conexão expira silenciosamente e, na próxima sessão, o SQLAlchemy tenta rollback numa conexão zumbi.
Solução:
python# Em database.py / engine setup — adicionar pool_pre_ping e pool_recycle
engine = create_async_engine(
    DATABASE_URL,
    pool_pre_ping=True,          # <-- testa a conexão antes de usar
    pool_recycle=1800,           # <-- descarta conexões com > 30min
    pool_size=5,
    max_overflow=10,
)
pool_pre_ping=True é o fix principal — ele emite um SELECT 1 antes de cada checkout do pool, detecta conexões fechadas e as descarta antes que o SQLAlchemy tente usá-las.

🟡 PROBLEMA 2 — /api/live/log-stream recebendo 401 em loop automatizado (RUÍDO OPERACIONAL)
O que é: 184 requisições com status 401 para /api/live/log-stream, exatamente a cada 60 segundos, das 09:00 às 12:06 (3h06min), vindas de 3 IPs AWS:
IPHits13.222.141.2201183.80.235.16439100.53.247.12630
O referer é https://scalpyn.vercel.app/trading-desk/diagnostics.
Diagnóstico: O frontend na página /diagnostics tem um polling a cada 60s que chama a URL direta do Cloud Run (scalpyn-wm56dfqgta-uc.a.run.app) em vez da URL canônica do backend. O token JWT enviado está expirado ou inválido — daí o 401 persistente. Os IPs são de CloudFront/AWS (provavelmente Vercel Edge ou um cron job externo).
Dois sub-problemas:

O polling continua mesmo recebendo 401 — não há lógica de back-off ou abort.
A URL hardcoded aponta para o Cloud Run diretamente em vez de usar a variável de ambiente.

Solução no frontend:

🟡 PROBLEMA 3 — /ws/alerts retornando 403 (autenticação WebSocket)
O que é: 6 hits com status 403 para /ws/alerts às 12:08–12:09Z, vindos do mesmo IP IPv6 2804:7f0:96c1:415f:... (parece IP brasileiro — possivelmente você mesmo testando).
Diagnóstico: O WebSocket está sendo rejeitado com 403 (Forbidden, não Unauthorized). O 403 difere do 401 — o token pode existir mas não tem permissão, ou o handshake WebSocket não está enviando o token corretamente no header.
WebSockets no browser não suportam headers customizados — o token JWT precisa ser enviado via query param ou via primeiro mensagem após conexão.
Solução:
python# No FastAPI, aceitar token via query param para WebSocket
@router.websocket("/ws/alerts")

    user = verify_token(token)  # validar aqui
    if not user:
        await websocket.close(code=4001)
        return
    await websocket.accept()
    ...
typescript// No frontend
const ws = new WebSocket(
  `wss://api.scalpyn.com/ws/alerts?token=${getAccessToken()}`

O plano está bem estruturado e tecnicamente consistente. Ele resolve corretamente o problema central: transformar o /dashboard/performance em um centro operacional real, e não apenas um painel visual.

Os pontos mais fortes:

separação clara entre observabilidade operacional e lógica de trading
proteção explícita das tasks críticas
uso correto de cache TTL para evitar travamento com celery.inspect()
foco em leitura/read-only
definição objetiva de SLO operacional
definição clara de estados críticos
validação explícita contra SQL real
preocupação com p95 dos endpoints
reutilização da arquitetura existente

A arquitetura proposta está madura para produção.

Pontos que eu ajustaria antes da implementação:

1. ALERTS DEVERIAM SER CENTRALIZADOS EM UM ENGINE

Hoje o /api/dashboard/alerts “chama internamente os outros”.

Isso funciona inicialmente, mas cria:

fan-out de requests
duplicação de lógica
latência acumulada
cascata de timeout
múltiplos hits Redis/Celery

Melhor:

DashboardHealthService
 ├── get_ingestion_health()
 ├── get_redis_health()
 ├── get_celery_health()
 ├── get_db_health()
 └── build_alerts()

O endpoint /alerts deveria consumir snapshots já carregados.

Senão haverá:

alerts
 -> celery
 -> redis
 -> db
 -> pipeline

e o frontend também chamando os endpoints individuais.

Resultado:
duplicação operacional.

2. CELERY INSPECT É O MAIOR RISCO DO PLANO

Esse é o ponto mais perigoso.

celery.control.inspect() pode:

bloquear
congelar worker
demorar vários segundos
falhar silenciosamente
gerar spike no broker

Especialmente em produção com Redis remoto.

Sugestão:

NÃO usar inspect a cada polling

Criar snapshot em background:

cached_celery_snapshot
updated_every=15s

Os endpoints apenas leem o snapshot.

Melhor ainda:

asyncio.create_task(refresh_celery_snapshot())

Senão o dashboard pode virar a origem da degradação.

3. REDIS INFO É CARO

INFO completo é relativamente pesado.

Não chamar múltiplas vezes.

Separar:

INFO stats
INFO memory

Evitar:

INFO ALL

E colocar TTL mínimo:

15s ideal
não 5s
4. DEFINIR PRIORIDADE DOS ALERTAS

Hoje há apenas level.

Sugiro:

{
  "severity": "critical",
  "category": "ingestion",
  "code": "worker_offline",
  "impact": "pipeline halted"
}

Porque futuramente isso permitirá:

alert routing
Slack/Discord
PagerDuty
webhook
métricas históricas
5. FALTA HISTÓRICO OPERACIONAL

O plano mostra apenas estado atual.

Isso é insuficiente para troubleshooting.

Sugestão:

Adicionar ring-buffer em memória

Últimos:

100 alerts
100 worker failures
100 Redis degradations

Mesmo sem persistência.

6. SCORE ENGINE PANEL PRECISA DIFERENCIAR:

Hoje o plano mistura:

score throughput
decisões
qualidade do score

Sugiro separar:

Throughput
scores/min
decisions/min
Quality
avg confidence
reject ratio
missing indicators
stale indicators
Distribution
histogram
p50/p95 score
7. PIPELINE LATENCY ESTÁ SUBESPECIFICADO

Esse ponto ainda está perigoso.

Hoje existem duas possibilidades:

(a) Prometheus parse
(b) SQL derivado

Mas elas medem coisas diferentes.

Recomendo formalizar:
Ingestion latency
NOW() - last candle
Decision latency
decision.created_at - candle.time
Processing latency
pipeline_end - pipeline_start

Senão o dashboard mostrará métricas semanticamente inconsistentes.

8. CUIDADO COM O FRONTEND POLLING

Atualmente:

alerts 15s
celery 30s
redis 30s
score 60s
latency 60s

Com múltiplos usuários simultâneos isso pode amplificar.

Sugestão:

Backend aggregation endpoint
/api/dashboard/overview

retornando:

{
  "alerts": {},
  "redis": {},
  "celery": {},
  "pipeline": {}
}

O frontend faria:

1 request
não 6

Isso reduz:

carga
fan-out
inconsistência temporal
9. FALTA PROTEÇÃO CONTRA QUERY PESADA

Endpoints de agregação precisam obrigatoriamente:

LIMITAR JANELAS

Exemplo:

WHERE created_at > NOW() - INTERVAL '24 HOURS'

e índices obrigatórios:

(created_at)
(symbol, created_at)
(decision, created_at)

Senão o dashboard degrada o banco.

10. DB HEALTH PRECISA MEDIR:

Não apenas:

SELECT 1

Mas:

checkout wait time
pool exhaustion
reconnect count
invalidated connections

Porque SELECT 1 pode passar mesmo com starvation do pool.

11. FALTA DEFINIÇÃO DE FAIL-SAFE

Se Redis cair:

dashboard quebra?
endpoint retorna 500?
frontend trava?

Todos endpoints precisam retornar:

{
  "status": "degraded",
  "error": "redis_timeout"
}

e nunca exception não tratada.

12. IMPORTANTE — NÃO USAR MÉTRICAS “AO VIVO” COMO SOURCE OF TRUTH

O dashboard deve ser:

eventually consistent

e não:

real-time blocking

Nunca bloquear request esperando Celery/Redis responder.

CONCLUSÃO

O plano está forte e pronto para execução.

Risco técnico principal:

Celery inspect
excesso de polling
múltiplas agregações em tempo real

Melhoria arquitetural mais importante:

Criar camada central:
OperationalSnapshotService

com:

cache
TTL
snapshots
agregações
alert engine

e os endpoints apenas consumindo snapshots.

Isso tornará:

o dashboard estável
barato operacionalmente
resiliente
escalável
seguro para produção

Sem essa camada, o dashboard corre o risco de se tornar um consumidor agressivo da própria infraestrutura que deveria monitorar.

# REFACTOR ARQUITETURAL — SEPARAÇÃO DE ESTÁGIOS DO PIPELINE E REMOÇÃO DA SOBRECARGA SEMÂNTICA DE is_approved

## CONTEXTO

O sistema atualmente utiliza um único campo:

```python
is_approved

para múltiplas responsabilidades diferentes:

símbolo monitorado
símbolo elegível para ingestão
símbolo promovido no pipeline
símbolo elegível para score
símbolo elegível para operação
símbolo apto para compra/venda

Isso criou ambiguidade semântica e possivelmente um deadlock lógico:

INGESTÃO
→ gera indicadores
→ promove L1/L2/L3
→ define is_approved

mas atualmente a ingestão parece depender de:

WHERE is_approved = true

o que cria dependência circular.

Além disso:

o sistema possui pipeline explícito:
POOL → L1 → L2 → L3
a UI confirma que “Approved” é estado do L3
portanto is_approved NÃO representa apenas “ativo presente no pool”
OBJETIVO DO REFACTOR

Separar claramente:

Monitoramento
Progressão no pipeline
Elegibilidade operacional
Elegibilidade de trade

Eliminando a sobrecarga semântica do campo único is_approved.

NOVA MODELAGEM DE DOMÍNIO
Arquitetura desejada
Ativos no POOL

Todo ativo presente no pool:

manualmente
ou
automaticamente

deve ser considerado:

is_monitored = true

Esses ativos:

DEVEM ser coletados
DEVEM gerar indicadores
DEVEM participar do pipeline

independentemente de aprovação operacional.

PROGRESSÃO DO PIPELINE
L1

Ativos monitorados são processados pelo filtro L1.

Se aprovados:

is_approved_l1 = true
L2

L2 consome:

is_approved_l1 = true

Se aprovado:

is_approved_l2 = true
L3

L3 consome:

is_approved_l2 = true

Se aprovado:

SPOT:
is_tradeok_spot = true
FUTURES LONG:
is_tradeok_long = true
FUTURES SHORT:
is_tradeok_short = true
IMPORTANTE

O coletor NÃO deve depender de:

is_tradeok
L1
L2
L3

A coleta deve depender apenas de:

is_monitored = true

ou equivalente semântico do POOL.

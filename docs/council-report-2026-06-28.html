Phase 3: Como implementar o pipeline de calibração de profiles?
2026-06-28  ·  5 advisors  ·  5 reviewers  ·  Chairman synthesis
Questão submetida ao conselho O audit confirmou que Phase 3 do pipeline Profile Intelligence (shadow validation → aplicar calibração ao config do profile) nunca foi implementada. 1.784 ajustes stuck em PENDING_VALIDATION; constraint do banco bloqueia auto-apply; current_value=null em 100% das sugestões; todos os diffs idênticos (buy threshold 65→70, default +5). Qual das 4 opções (A: UI manual, B: ciclo automático, C: híbrido, D: repensar primeiro) é a abordagem correta para implementar Phase 3?
Veredicto do Chairman

Onde o conselho concorda

Todos os advisors que analisaram a evidência chegaram ao mesmo diagnóstico: os 1.784 registros não são um backlog a processar — são artefatos de um sistema que nunca funcionou corretamente. Dois bugs os tornam irrecuperáveis no estado atual: current_value=null significa que não existe baseline, e o diff uniforme de +5 significa que nenhuma análise foi executada. Opções A, B e C pressupõem inputs que não existem. D não é uma opção — é um pré-requisito.

O Executor foi eleito o mais forte por todos os 5 revisores porque foi o único que tratou D como precondição e então nomeou o que construir depois. O Expansionista foi eleito o mais perigoso por todos os 5 porque "construção concorrente" é uma corrida contra uma fundação quebrada.

Onde o conselho diverge

Sequenciamento vs. paralelismo. O Expansionista argumenta que corrigir o sinal e construir o ciclo de auto-apply podem ocorrer em paralelo. Todos os outros advisors — e todos os 5 revisores — discordam. A dependência é estrutural: run_apply_cycle() não pode executar um gate de win_rate quando a baseline é null.

Option C vs. parar no D. Contrarian e First Principles dizem que D é a resposta completa — não arquitetar Phase 3 até que sinal válido exista. O Executor diz que C é o destino correto após os dois bugs serem corrigidos. O desempate vem do insight da constraint do banco: Option C já está arquitetada no schema (mutation_applied=false OR requires_human_approval=true). Option B requer uma migração para alterar essa constraint. O schema resolveu este debate antes do conselho se reunir.

Destino dos 1.784 registros corrompidos. Nenhum advisor endereçou isso explicitamente. Os revisores identificaram. É uma questão operacional live que deve ser resolvida antes de re-executar a análise.

Blind spots que o conselho capturou

Três gaps, todos identificados pelos revisores, todos mais urgentes do que o debate arquitetural:

1. O generator ainda está rodando (crítico). Ninguém perguntou se o INSERT quebrado está gerando novos registros corrompidos agora. Se o ciclo de sugestões está ativo, o número 1.784 está crescendo. Cada etapa "aguardar próximo ciclo" em qualquer plano popula com dados novos inválidos. Isto deve ser a primeira ação.

2. Volume de shadow trades por profile é desconhecido. 32 profiles, nenhum live. Se cada profile tem poucos shadow completions, nenhum gate estatístico pode atingir significância independentemente da qualidade do sinal. Toda a viabilidade do win_rate gate depende desse número.

3. Shadow monitor estava quebrado até 2026-06-27 (crítico de domínio). Zero completions de shadow trades de 25/06 a 27/06 por causa de um NoReferencedTableError em ml_opportunity_rankings. Mesmo após corrigir os dois bugs, pode não haver shadow trades fechados suficientes para qualquer gate estatístico durante semanas. O repair do shadow monitor em 27/06 inicia o relógio — não o zera.

A Recomendação

Construir Option C, em sequência estrita, começando com um stop.

O destino arquitetural é Option C: validar automaticamente um gate de performance, então exigir confirmação humana de um clique antes de escrever no profile config. As razões são concretas:

• A constraint do banco já está pré-configurada para Option C. Construir Option B requer migração de schema e remove uma camada de segurança deliberadamente projetada.
• Option A escala para 32 profiles agora mas falha com 300. Option B remove supervisão humana antes que o sistema tenha demonstrado que funciona.
• Option C dá ao sistema um ciclo correto sob supervisão humana antes de ganhar autonomia. Esse é o prior correto dado que Phase 3 nunca foi entregue.

Mas Option C não pode ser iniciada até que quatro condições sejam verdadeiras: (1) o job de geração de sugestões está pausado; (2) o bug INSERT está corrigido; (3) o diff generator produz deltas por-profile a partir de performance real; (4) os 1.784 registros corrompidos estão explicitamente tombstoned ou purgados.

Após essas quatro condições: deixe o ciclo shadow rodar com lógica correta. Antes de implementar run_apply_cycle(), verifique se shadow trades fechados com outcome suficientes existem por profile. Dado que o shadow monitor só voltou em 27/06, a espera honesta pode ser 2–4 semanas. Isso é aceitável — os 32 profiles não são live. Não há emergência que justifique construir Phase 3 sobre inputs quebrados.

A única coisa a fazer primeiro

Pausar o job de geração de sugestões. Não corrigir o bug INSERT. Não projetar a UI de aprovação. Não purgar os 1.784 registros. Parar o sangramento primeiro. Toda ação subsequente — correções de bugs, arquitetura, acumulação de dados — é comprometida se o ciclo quebrado continua escrevendo registros uniform-diff, null-baseline em PENDING_VALIDATION enquanto você trabalha.

Respostas dos advisors
■ The Contrarian Fix First

Options A, B, and C all share the same fatal assumption: that the 1,784 pending suggestions contain signal worth acting on. They don't. They're uniform noise from a broken generator.

Every single diff is buy threshold 65→70. Not "most" — all 1,784. That's not a calibration result, that's a hardcoded default masquerading as analysis. And current_value=null means you don't even have a real baseline — you can't compute a diff you don't have.

The specific danger with Option B (auto-apply): you build a statistical gate comparing win_rate before/after, but the "after" configuration is "buy threshold was bumped +5 uniformly." If win_rate improves, is that the calibration working or market conditions shifting? You cannot answer that question because the intervention wasn't personalized — it was noise.

Option D is the only honest answer, but the framing is too gentle. "Rethink first" implies the signal is weak. The reality is the signal is absent. You're not choosing between implementation approaches — you're choosing whether to build on concrete or on sand.

Fix the INSERT bug. Fix the diff generator to produce profile-specific deltas. Purge or tombstone the corrupt 1,784 rows. Then design Phase 3. In that order, no exceptions.

■ The First Principles Thinker Fix First

The question "how to implement Phase 3" contains a fatal embedded assumption: that Phase 1 and 2 produced valid signal worth applying. They didn't.

1. Uniform diffs. 1,784 records, 32 distinct profiles, zero variation in the recommendation. A calibration system that produces identical output regardless of input isn't calibrating — it's broadcasting a hardcoded guess. The analysis isn't broken in a subtle way; it never ran.

2. current_value=null. The diff comparison requires knowing the current value to compute a meaningful delta. It was never read. Every suggestion is a diff between null and a constant. This isn't a minor bug — it means no suggestion in the database reflects actual profile state.

Given this, Options A, B, and C are all wrong. They disagree only on who pulls the trigger on garbage data. Option C (hybrid) is the most dangerous because it creates the illusion of rigor — a statistical gate passing recommendations that were never computed correctly.

The real first-principles question: What does a profile miscalibration actually look like in this system? What observable metric signals that a threshold is wrong, and by how much? Until that's defined and the analysis actually computes it per-profile, Phase 3 has no inputs.

Invalidate the 1,784 records. Rebuild the analysis from the measurement question. Phase 3 is premature.

■ The Expansionist Rejected by peer review

Option D is a trap disguised as prudence. Fixing the signal first means Phase 3 never ships — it becomes the permanent excuse. Meanwhile 32 profiles sit uncalibrated indefinitely.

Here's what everyone is undervaluing: the 1,784 stuck records aren't just a bug backlog — they're a labeled dataset. Each pending adjustment has a corresponding shadow trade history. That's a retrospective ground truth you can mine right now to validate whether the calibration logic even moves the needle.

Push for Option B, with proper signal computation from day one. The real upside: if run_apply_cycle() auto-calibrates 32 profiles and shadow performance measurably improves, you have proof-of-concept for a self-tuning system — every live profile continuously recalibrated against its own shadow twin.

Option C is the right political answer, wrong product answer. Build B with a 30-day kill switch instead.

Note from peer review: The "labeled dataset" framing collapses because current_value=null means no "before" state exists. The statistical gate cannot execute. All 5 reviewers rated this response as having the biggest blind spot.

■ The Outsider Fix First

Stop. Before any implementation debate, look at what you actually have: 1,784 records where current_value=null and every single diff is identical. You don't have a Phase 3 problem. You have a data integrity problem dressed up as an architecture decision.

The builders are asking "how do we build the approval system?" when the honest question is "approve what, exactly?" You cannot validate performance of an adjustment you cannot measure — because you don't know what value you started from. And if every diff is +5 regardless of profile, the algorithm isn't analyzing profiles at all; it's applying a hardcoded constant masquerading as intelligence.

This is the curse of knowledge in action: everyone in the room knows what the system should be doing, so they're debating the next step. A newcomer reads the evidence and asks: has this system ever worked?

The correct sequence: fix the INSERT bug, let the system generate real diffs across profiles (they should differ), then verify the analysis logic actually produces differentiated recommendations before choosing an approval model.

■ The Executor Strongest — rated by all 5 reviewers

Option D isn't a choice — it's a prerequisite. But frame it right: it's two bug fixes, not a rethink.

Monday morning, do these two things first:

  1. Find the hardcoded INSERT that sets current_value=null and fix it. This is a single line change. Without it, you cannot calculate any before/after delta.
  2. Find where the uniform 65→70 bump is generated and fix the analysis to derive per-profile diffs from actual shadow performance data.

After those two fixes, implement Option C. Here's why it's the fastest path: your DB constraint (mutation_applied=false OR requires_human_approval=true) is already pre-wired for Option C. Implementing Option B requires a schema migration to drop or modify that constraint. Option C just needs: a run_apply_cycle() that sets requires_human_approval=true on records that clear the performance gate, plus a minimal approval UI — one table, one button per row, one POST endpoint.

Concrete sequence:

  1. Fix current_value=null bug (today)
  2. Fix uniform-diff analysis (today/tomorrow)
  3. Let shadow data regenerate with correct diffs (wait for next cycle)
  4. Implement run_apply_cycle() with win_rate gate
  5. Build minimal one-click approval UI

Total: 3–4 days of actual build work, preceded by data accumulation. Acceptable cost versus shipping Phase 3 on broken signal data.
